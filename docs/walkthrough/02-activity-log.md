# 活动日志页用户指南

日志页把器官变更翻译成按日期排列的活动时间线。它面向的是复盘场景：你不一定要处理某一条风险卡片，只是想知道“最近 Agent 系统在自己的器官上发生了什么”。

相比看板页，日志页更像一份自动生成的日记。它把新增、更新、删除、上游同步等动作按天聚合，让非工程视角也能读懂变化节奏。

## 1. 看一天发生了什么

点击顶部“日志”进入活动时间线。你可以选择全部时间、近 30 天或近 7 天。

![活动时间线](assets/03-activity-timeline.png)

每个日期卡片都在回答三个问题：

- 这一天一共有多少处改动。
- 这些改动主要是新增、更新还是删除。
- 它们集中在哪些目录、skill 或系统里。

比如截图中，`2026-07-13` 这一天有 1 处更新，摘要落在 `cron`。用户不需要先知道账本格式，也能明白这天的变化重点：计划任务相关配置被更新过。

## 2. 展开当天明细

当某一天看起来值得关注时，点击“查看当天逐条改动”，右侧会打开当天明细。

![当天明细抽屉](assets/04-activity-day-drawer.png)

明细列表的作用是把“这一天有变化”继续拆成“具体哪些地方变了”。它会展示文件路径、操作类型和所属系统，让你决定下一步该去看哪条记录。

如果当天改动很多，这里会比翻终端日志更容易读：你可以先看聚合摘要，再进入具体条目。

## 3. 复制当天简报

当天明细里的“复制当天简报给 Coding Agent”按钮（见上图），是给进一步分析准备的。你可以把当天所有变更整理成一段说明，交给 Coding Agent 或同事继续排查。

适合使用简报的情况：

- 某一天的改动突然变多。
- 改动集中在高敏感目录，例如 skills、cron、memory。
- 你想让 Coding Agent 帮你回到仓库里看真实 diff。
- 你要把当天变化同步给团队，但不想手动整理每个路径。

这让日志页不只是“看历史”，也成为复盘和交接的入口。

## 4. 为什么日志页不显示文件内容

日志页只展示“发生了什么”和“发生在哪里”。它不会展示文件正文、diff 或密钥内容。

这是为了让用户可以放心地用日志页做日常复盘和截图沟通。页面提供的是足够定位问题的线索，而不是把敏感内容直接摊开。

如果你需要判断某个变更的语义是否正确，可以从日志进入看板详情，再把简报交给 Coding Agent 或回到本机 git 查看内容。

## 5. 推荐用法

日常可以这样用：

1. 每天或每周打开日志页，看改动数量是否异常。
2. 发现某天变化集中，就展开当天明细。
3. 先判断是正常上游同步、日常维护，还是异常本地改动。
4. 对可疑日期复制简报，进入代码级检查。
5. 如果确认需要处置，再回到看板页查看单条变更并执行后续命令。

日志页的价值，是把“不可读的变更流水”变成“用户能复盘的活动记录”。
