## 任务 1（🔴 BLOCKER）：创建 `.gitattributes` 并重正规化

**为什么**：无 `.gitattributes`，Windows 默认 `core.autocrlf=true` 会在 checkout 时把 LF 换成 CRLF。之后 `fileSha()`（`util.ts:206`）对工作区文件算的 hash 是 CRLF 版本，而 `gitShowHead()` 取的是 HEAD 里的 LF 版本 → `before_hash != actual` → **哈希链断裂**。测试侧靠 `core.autocrlf=false` 躲过了，但真实用户 clone 会中招。

**定位**：`ls /c/Hello-World/organledger/.gitattributes`（应不存在）

**动作**：在仓库根新建 `.gitattributes`：

```gitattributes
# 钉死行尾为 LF，保证 sha256 哈希链跨平台字节一致。
* text=auto eol=lf

# 源码与数据显式 LF
*.ts   text eol=lf
*.js   text eol=lf
*.py   text eol=lf
*.json text eol=lf
*.jsonl text eol=lf
*.md   text eol=lf
*.yml  text eol=lf
*.css  text eol=lf
*.html text eol=lf

# 二进制：禁止任何换行/文本转换
*.png  binary
*.ico  binary
*.pyc  binary
*.woff binary
*.woff2 binary
```

**然后重正规化已入库文件**：
```bash
cd /c/Hello-World/organledger
git add .gitattributes
git add --renormalize .
git status        # 查看被重正规化的文件
```
若 `git status` 显示大量文件被改，说明工作区之前确有 CRLF——这正是要修的。提交它（P6 阶段统一 commit，或此处单独 commit 一次 `chore: pin LF via .gitattributes`）。

**验收**（自检）：
- [ ] `.gitattributes` 存在且含 `eol=lf`
- [ ] `git add --renormalize .` 后无报错
- [ ] `node --test test/*.test.ts` 仍全绿（正规化不应破坏测试）

## 任务 2（🔴 BLOCKER）：修 `backfill.ts` 及同类 CRLF 解析

**为什么**：`git show --raw` 在 Windows 上可能输出 CRLF。`body.split("\n")` 只切 `\n`，每行尾残留 `\r`，导致后续 `indexOf("\t")`、`split(/\s+/)` 的 `status`/`repoRel` 字段带 `\r` → 回填的 before/after hash 错位。

**定位**：
```bash
grep -rn 'split("\\\\n")' src --include=*.ts
grep -rn "split('\\\\n')" src --include=*.ts
```

**主改点** `src/onboard/backfill.ts`（审计时约 line 190）：
```diff
- for (const line of body.split("\n")) {
+ for (const line of body.split(/\r?\n/)) {
```

**逐个评估其它命中项**（判定规则）：
- ✅ 改成 `/\r?\n/`：**解析「外部命令输出」或「读自磁盘的文本文件」**的 split（git 输出、文件内容、用户粘贴）。
- ⛔ 不动：**切割「自己刚拼出来的、确定用 `\n` 的字符串」**（如 `manualWindowsSteps` 里 `.message.split("\n")[0]` 取错误首行——这是内存字符串，可留；但若是解析 stderr 建议也改，稳妥起见改成 `/\r?\n/` 无害）。
- 参考安全基线：`util.ts:168 readJsonl` 与 `util.ts:212 countLines` **已经**用 `/\r?\n/`，是正确范式，照抄。

**验收**：
- [ ] `backfill.ts` 已用 `/\r?\n/`
- [ ] grep 复查：所有解析外部/文件文本的 split 已改
- [ ] `node --test test/attribution.test.ts test/provenance.test.ts test/core.test.ts` 全绿

## 任务 3（🟡 MAJOR）：Windows SIGBREAK 优雅关闭

**为什么**：Windows 不支持 `SIGTERM`。现有 `cli/index.ts:348-349` 注册了 SIGINT+SIGTERM，但 Windows 场景（计划任务停止、Ctrl-Break）无法干净触发 `shutdown()` → 不释放 daemon.lock → 残留锁。

**定位**：`grep -n "SIGINT\|SIGTERM\|process.on" src/cli/index.ts`

**改点**（`src/cli/index.ts`，shutdown 注册处）：
```diff
  process.on("SIGINT", shutdown);
  process.on("SIGTERM", shutdown);
+ // Windows 无 SIGTERM；SIGBREAK 覆盖 Ctrl-Break 与计划任务停止路径。
+ if (process.platform === "win32") process.on("SIGBREAK", shutdown);
```

> 说明：`shutdown` 若可能被多次触发，确认其幂等（现有实现里 `releaseLock` 用 try/catch 兜底，且 `stopped` 标志防重入——**先读一眼 `daemon.ts:98-105` 确认幂等，若不幂等，在 shutdown 里加一次性 guard**）。

**验收**：
- [ ] 已加 SIGBREAK（win32 分支内）
- [ ] shutdown 幂等（读 daemon.ts 确认或加 guard）
- [ ] 全量测试仍绿（此改动无测试覆盖属正常，别硬造）

## 任务 5（🟡 MINOR）：reveal UNC 路径注释 + xdg-open 防御

**为什么**：`reveal.ts:55` 已用 `path.isAbsolute` 兜底 UNC，但意图不清晰。属加固+可读性，非漏洞。

**改点**（`reveal.ts` 越界检测处）：
```diff
- if (path.isAbsolute(norm) || /^[a-zA-Z]:/.test(norm)) {
+ // 拦截绝对路径、盘符(C:)、UNC(\\server\share)——防止相对路径逃逸出目标 home。
+ if (path.isAbsolute(norm) || /^[a-zA-Z]:/.test(norm) || norm.startsWith("\\\\")) {
    return { ok: false, status: 403, error: "out of bounds" };
  }
```
> 注意 `norm` 在该函数里已把 `\`→`/` 正规化过吗？**先读 reveal.ts:47-70 确认 `norm` 的实际形态**——若已正规化为 `/`，UNC 会是 `//`，则检测 `norm.startsWith("//")`。**以实际为准**，这是本手册最容易踩错的地方。

**xdg-open**（`cli/index.ts:65`）：仅与 Linux 相关，Win/Mac 无关。时间紧可**跳过**（记 TODO）。若做：加注释说明「Linux 最小系统可能无 xdg-open，失败静默（detached+stdio:ignore 已兜底）」。
