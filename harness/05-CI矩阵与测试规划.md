# 05-CI 矩阵与测试规划（任务 6）

## 目标
把 `.github/workflows/ci.yml` 从「仅 ubuntu」升级为 **ubuntu + windows + macos** 三平台矩阵，让本次修复的跨平台行为被 CI 持续守护。

## 现状
```yaml
check:
  runs-on: ubuntu-latest
  ...
  node-version: "24"
```
`grep -n "runs-on\|matrix\|node-version" .github/workflows/ci.yml` 确认现场。

## 目标写法（示意，以现场 job 结构为准）
```yaml
jobs:
  check:
    strategy:
      fail-fast: false          # 一个平台挂不要掐掉其它平台，方便定位
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "24"
      # 若有 npm ci / 依赖安装步骤，保留
      - run: npm ci
      # typecheck（若原 ci 有）
      - run: npm run typecheck
      # 测试
      - run: node --test test/*.test.ts
```

## 关键注意点
1. **保留原 job 里已有的所有步骤**（checkout / setup-node / npm ci / typecheck），只在其上包一层 matrix + 改 `runs-on`。**读现场文件，别整段替换丢步骤。**
2. **glob 展开**：`node --test test/*.test.ts` 在 Windows 的 `cmd`/`pwsh` 下 `*.test.ts` 可能不被 shell 展开。
   - 稳妥做法：改用 Node 自身的 glob——`node --test --test-name-pattern` 不解决这个；更稳的是 `node --test test/`（Node ≥ 20 的 `--test` 可直接接目录，自动发现 `*.test.*`）。**验证 Node 24 是否支持接目录**（支持）。或在 `package.json` 的 `test` script 里用可跨平台展开的方式，然后 CI 跑 `npm test`。
   - 推荐：CI 里用 `npm test`，并确保 `package.json` 的 `test` script 跨平台可用（若 `node --test test/*.test.ts` 在 Windows 本机能跑通——本机就是 Windows，P-1 基线已验证——那 glob 已 OK，可直接用）。
3. **python 依赖**：GitHub 的 windows/macos runner 默认带 python，但 hermes 测试已在任务 4 改为「缺 python 则 skip」，双保险。
4. **`.gitattributes` 生效**：checkout 时 attributes 已生效，CI 上工作区为 LF，哈希链测试应在三平台一致通过——**这正是三矩阵要证明的核心**。

## 验收
- [ ] `ci.yml` 有 `matrix.os` 三平台 + `runs-on: ${{ matrix.os }}`
- [ ] 保留了原有全部步骤
- [ ] 本机（Windows）先手动模拟 CI 命令跑一遍：`npm run typecheck && node --test test/*.test.ts` 全绿
- [ ] （推送后）三平台 CI 全绿 —— 此项由人类 push 后在 GitHub 观察，执行者把「待观察」记入 TODO
