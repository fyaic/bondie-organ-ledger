# 03-平台适配决策树

## 决策符号
| 符号 | 含义 | 你的行为 |
|------|------|---------|
| 🟢 保留/新增 | 直接实现，两平台原生可行 | 按 04 手册实现 |

## 决策 1：换行 / 哈希链（BLOCKER）

| 特性 | 现状 | 决策 | 目标方案 | 验收 |
|------|------|------|---------|------|
| 全仓库换行策略 | 无 `.gitattributes` | 🟢 新增 | 建 `.gitattributes`：`* text=auto eol=lf`，二进制 `-text`（png/ico/pyc）。见 04-任务1 | Win clone 后所有文本文件为 LF |
| 已入库文件 CRLF 残留 | 未知 | 🟡 适配 | 加完 attributes 后 `git add --renormalize .` 重正规化并提交 | `git diff` 无杂散 CRLF |
| `backfill.ts` git raw 解析 | `split("\n")` | 🟡 适配 | 改 `split(/\r?\n/)`（见 04-任务2） | Windows git 带 CRLF 也能正确解析 |
| 其它裸 `split("\n")` | 需扫描 | 🟡 适配 | grep 全仓，逐个评估：**解析外部/文件文本的**改 `/\r?\n/`；**拼接自己输出的**不动 | 见 04-任务2 扫描清单 |

> ⚠️ 红线：**不要**去 `fileSha`/`canonicalJson` 里“正规化换行”。正确修法是**从源头（.gitattributes）杜绝 CRLF 进入工作区**，而不是在哈希时改字节——后者会改变账本语义、破坏与既有账本的兼容。

## 决策 6：明确不做（⏳ 二期）

| 特性 | 决策 | 理由 |
|------|------|------|
| mac/Linux 自启动生产化（launchd/systemd 自动安装 + 真机验证） | ⏳ 二期 | 超出「代码级加固」范围；`autostart.ts` 现状（输出模板 + 手动指引）本次保持不变。**不要动 `autostart.ts`。** |
| 改造账本序列化 / canonicalJson | 🔴 禁止 | 会破坏既有账本兼容与哈希链语义 |
| 引入 cross-platform 第三方库（如 `cross-spawn`） | 🔴 禁止 | 违反「无新依赖」约束，且现有分派已正确 |
